6.4. Comparing/Isolating/Testing

In this section you will learn to:
  • Compare performance or configuration differences with other systems or baseline
  • Perform root-cause analysis
  • Verify the fix to an isolated problem

Comparing

It is very useful to compare a broken system with a similar system that is functioning properly. The systems to be compared should be as close to being identical as possible. This includes hardware configuration, operating system configuration, and installed applications. Do not make any adjustments to the healthy system, otherwise you may have two systems you will have to troubleshoot instead of one.
Compare system logs and program output between the two systems. Sometimes this will help locate relevant error messages more quickly. Once the errors have been identified, the problem can be corrected then tested.
Another point of comparison are configuration files. Make copies of the /etc directory from both systems and run a recursive diff to identify differences. This process is a very efficient way to pinpoint configuration information that may need more detailed examination.
Performance monitoring tools, such as sar and top allow machine performance to be compared between two systems. sar will allow for comparison of metrics taken over a long period of time while top allows currently active processes to be compared.
Finally, another possibility is to compare two machines that have the same software installed and congruent configurations, but have a different hardware platform. This will identify problems where the hardware is overextended to run software beyond its capacity.

Isolating

A common troubleshooting mistake is to take the first identified cause of a problem and immediately take steps to address that problem. This can lead to a solution to a problem that never existed.
A better approach is to brainstorm as many causes as possible that might produce the symptoms that a given broken system displays. The goal of troubleshooting is to solve the root cause of a problem rather than simply addressing symptoms.
Once all possible causes have been identified, rank them by difficulty to troubleshoot and solve. Start with easiest causes first then work down the list to the most complex problems to solve. This approach to troubleshooting is more efficient because it eliminates the easier problems to diagnose first, rather than spend a lot of time troubleshooting a complex problem that is nonexistent.

Testing

When troubleshooting, make one change at a time then test the impact on the performance of the system. Making more than one change at at time adds more variables of complexity to the troubleshooting process. For example two changes may cancel each other out when making the changes one at a time would have clearly distinguished between the corrective modification and the detrimental one.
